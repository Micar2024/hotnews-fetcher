# 别让 AI 自己审自己：我用 GitHub + ChatGPT Plus 跑通了一套协作链路

故事是这样的。

前段时间我在用 AI 编程工具写一个功能。写完了，我想找个东西帮我 Review 一下代码。你猜怎么着，我让写代码的那个 AI 自己审自己。

它当然说没问题。

上线三天后暴露了一个 bug，最后复盘发现，是它引入的边界条件错误，它自己 Review 的时候完全没看出来。

后来我复盘这件事，得出一个判断。

> **AI Coding 最大的风险不是写得慢，而是没人独立验收。**

写代码的和审代码的是同一个 AI，那不就是自己出卷自己答自己批改吗。

![错误链路：AI 自己审自己](wx_img_1_error_chain.png)

这不是个例。很多团队用 AI 协作都会踩这个坑，多个 AI 在给建议，但没有统一事实源；任务靠口头描述推进，执行 Agent 容易扩大范围；代码由同一个 Agent 写、同一个 Agent 审，看似闭环，实际只是同一个盲区反复确认。

后来我们跑通了一套方案。核心就三件事，GitHub 做唯一事实源，任务包做标准输入，ChatGPT Plus 做独立 PM Review。

我一个个聊。

## 先说任务包

这是整套方案真正的起点。你跟 AI 说「你看着改一下这个功能」，这是最贵的一句话。任务包写得含糊，后面所有 Agent 都会在错误轨道上加速。

一个合格的任务包必须包含这些东西。目标，用一句话说清要做什么。允许范围，哪些文件可以改。禁止范围，哪些东西不能碰。验收命令，怎么证明做对了。回滚方式，出了问题怎么恢复。

> **任务包五条硬检查**
> 1. 目标能不能一句话说清？
> 2. 禁区写没写？
> 3. 验收标准有没有具体到测试或命令？
> 4. 引用的文件还在不在？
> 5. 验证方案是不是可执行？

执行 Agent 有权拒包。任务包不达标，退回要求补全。先做着再说，是最贵的废话。

## 然后是 GitHub 为什么必须是事实源

很多团队用 AI 协作的时候，事实源是聊天记录。你上次说要改这个，我没说过，这种对话每天都在发生。

聊天记录不是事实源。飞书文档不是事实源。甚至你脑子里的想法也不是事实源。

唯一能被多个 Agent 同时引用、不会产生歧义的事实源，是 GitHub。Issue 是任务入口，PR 是执行证据，Review 是审计记录，Merge 是授权动作。所有角色都从同一个地方读数据，往同一个地方写结果。

> **如果多个 Agent 在读不同版本的「事实」，它们越努力，结果越离谱。**

![正确链路：执行、复审、审计分开](wx_img_2_correct_chain.png)

## 协作分工

整套体系里一共有六层。

> **Codex｜执行层**
> 按任务包写代码、补测试、提交 PR。

> **Claude Code｜复审层**
> 看 diff、调用链、测试覆盖，输出 PASS / HOLD。

> **ChatGPT Plus｜审计层**
> 审边界、风险、证据、表达，做独立 PM Review。

> **OpenClaw｜总控层**
> 维护项目节奏、派发任务、收口回执。

> **Hermes Agent｜策略层**
> 复杂问题拆解、任务包设计和方法论沉淀。

> **人｜决策层**
> 确认目标、控制风险、决定是否合入。AI 可以执行、复审、审计，但结果必须有人承担。

你可能会问，需要这么多角色吗。

我可以直接告诉你：执行和复审不能是同一个 AI，审计和执行不能是同一个 AI。这是整套方案的底层逻辑。

如果写代码的和审代码的是同一个模型，那跟自己审自己没有任何区别。Claude 复审 Codex 的代码，ChatGPT 审整体逻辑，人做最终决策——每一层都不能跟上一层是同一个。

![正确链路：执行、复审、审计分开](wx_img_2_correct_chain.png)

## ChatGPT 审什么，不审什么

很多人误解了 ChatGPT 在这套体系里的角色。它不是来写代码的，也不是来跑测试的。它是一个独立的 PM Review 角色。

> **ChatGPT 审四件事：**
> - 审边界：改动有没有超出任务包范围
> - 审风险：已知风险有没有诚实披露
> - 审表达：有没有误导性承诺
> - 审证据：测试声明和 PR 证据是否对得上

> **ChatGPT 不审四件事：**
> - 不替代代码复审
> - 不验证本地运行结果
> - 不确认数据库或线上状态
> - 不批准 merge

一句话，ChatGPT 是审计员，不是执行者。它看的是这件事做对了没有，而不是代码怎么写。

## 标准流程怎么跑

> **标准流程六步**
> 1. 策略顾问把目标拆成任务包，写进 GitHub Issue。
> 2. 执行层读取 Issue，在 feature branch 内完成代码和测试，提交 PR。
> 3. 复审层检查 diff 和测试覆盖，输出 PASS 或 HOLD。
> 4. 如果 HOLD，执行层修复后重新提交。
> 5. 把 PR 链接发给 ChatGPT，让它做独立 PM Review。
> 6. 负责人验收，合入主分支。

这六步看起来简单，但每一步都有明确的输入输出和阻塞条件。第 3 步 HOLD 了，第 4 步才能启动。第 5 步 ChatGPT 指出风险，第 6 步负责人才决定要不要合入。

不是所有任务都需要走完六步。低风险任务可以缩短，但至少要保留：任务包 → PR → Review → Merge 这四步。

## 这不是工具测评

我试过很多 AI 编程工具。但工具不是重点。Codex 可以换，Claude 可以换，ChatGPT 可以换，OpenClaw 也可以换。

重点是流程。

> **只要满足这三个条件：**
> - GitHub 是唯一事实源
> - 执行和复审不是同一个 AI
> - 有独立审计层
> 任何工具组合都能跑起来。

关键不是工具，是流程设计和职责分层。

![最小闭环：低风险任务先跑通](wx_img_3_minimal_loop.png)

## 如果你想试一下

不需要一次到位。

> **最小闭环：**
> 1. 找一个低风险任务，写成任务包，发 GitHub Issue。
> 2. 让一个 AI 执行，提交 PR。
> 3. 把 PR 链接发给另一个 AI 做 Review。
> 4. 自己看一眼 Review 结果，决定是否合入。
> 这四步跑通一次，你就知道这套东西为什么值得继续投入了。

工具不重要，流程才重要。职责分清楚，比什么都重要。

AI Coding 不缺能力，缺的是一个清醒的流程。
