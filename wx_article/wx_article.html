别让 AI 自己审自己：我用 GitHub + ChatGPT Plus 跑通了一套协作链路

故事是这样的。

前段时间我在用 AI 编程工具写一个功能。写完了，我想找个东西帮我 Review 一下代码。你猜怎么着，我让写代码的那个 AI 自己审自己。

它当然说没问题。

上线三天后暴露了一个 bug，最后复盘发现，是它引入的边界条件错误，它自己 Review 的时候完全没看出来。

后来我复盘这件事，得出一个判断。

AI Coding 最大的风险不是写得慢，而是没人独立验收。

写代码的和审代码的是同一个 AI，那不就是自己出卷自己答自己批改吗。

这不是个例。很多团队用 AI 协作都会踩这个坑，多个 AI 在给建议，但没有统一事实源；任务靠口头描述推进，执行 Agent 容易扩大范围；代码由同一个 Agent 写、同一个 Agent 审，看似闭环，实际只是同一个盲区反复确认。

后来我们跑通了一套方案。核心就三件事，GitHub 做唯一事实源，任务包做标准输入，ChatGPT Plus 做独立 PM Review。

我一个个聊。

先说任务包

这是整套方案真正的起点。你跟 AI 说「你看着改一下这个功能」，这是最贵的一句话。任务包写得含糊，后面所有 Agent 都会在错误轨道上加速。

一个合格的任务包必须包含这些东西。目标，用一句话说清要做什么。允许范围，哪些文件可以改。禁止范围，哪些东西不能碰。验收命令，怎么证明做对了。回滚方式，出了问题怎么恢复。

任务包五条硬检查

1. 目标能不能一句话说清？

2. 禁区写没写？

3. 验收标准有没有具体到测试或命令？

4. 引用的文件还在不在？

5. 验证方案是不是可执行？

执行 Agent 有权拒包。任务包不达标，退回要求补全。先做着再说，是最贵的废话。

然后是 GitHub 为什么必须是事实源

很多团队用 AI 协作的时候，事实源是聊天记录。你上次说要改这个，我没说过，这种对话每天都在发生。

我们的做法是，所有任务、所有变更、所有审查结论，都必须落在 GitHub 上。

任务目标写在 Issue 里。代码变更在 PR 里。审查结论在 PR Review 里。验收证据在 PR Comment 里。

聊天可以讨论，但聊天不是事实源。GitHub 是。

这套做法不是为了流程好看，而是为了让每一次变更都有可追溯的输入、输出、审查和验收证据。

接下来聊协作分工

这套链路里，工具不是按品牌分工，而是按职责分工。谁负责拆任务，谁负责写代码，谁负责复审，谁负责审计，谁负责最终拍板，必须分清楚。

我的实际组合是这样。

Codex CLI｜执行层

按任务包写代码、补测试、提交 PR。

Claude Code｜复审层

看 diff、调用链、测试覆盖，输出 PASS / HOLD。

ChatGPT Plus｜审计层

审边界、风险、证据、表达，做独立 PM Review。

OpenClaw｜总控层

维护项目节奏、派发任务、收口回执。

Hermes Agent｜策略层

复杂问题拆解、任务包设计和方法论沉淀。

人｜决策层

确认目标、控制风险、决定是否合入。AI 可以执行、复审、审计，但结果必须有人承担。

这篇文章不是工具测评。Codex、Claude Code、OpenClaw、Hermes 都可以替换，真正不能替换的是职责分层、事实源和人工授权点。

ChatGPT 审什么，不审什么，这个边界必须讲清楚

一句话，ChatGPT 是审计员，不是执行者。它看的是这件事做对了没有，而不是代码怎么写。

举个真实的例子

我们做了一个热搜抓取脚本的 PR。执行 Agent 说测试通过，可以合并。

我没有直接合并，而是把 PR 链接丢给了 ChatGPT。它不去判断每一行代码优不优雅，它问了四个问题。

第一，README 里有没有依赖安装和运行命令？第二，PR 里说测试通过，但后面又补了新测试，PR body 没同步更新，测试声明和实际变更已经对不上了。第三，外部 API 失败时有没有回退机制？第四，代码有没有把 cookie、token、私密请求头写进去？

四个问题里，两个是阻塞项。为什么？因为它们不一定会导致代码立刻报错，但会导致后续接手的人无法复现、无法判断风险，也无法确认这次变更到底有没有被完整验收。修复之后才合入。

这就是 PM Review 的价值。它不是帮你找 bug，而是帮你确认这件事做对了没有。

然后是标准流程

标准流程六步

1. 策略顾问把目标拆成任务包，写进 GitHub Issue。

2. 执行层读取 Issue，在 feature branch 内完成代码和测试，提交 PR。

3. 复审层检查 diff 和测试覆盖，输出 PASS 或 HOLD。

4. 如果 HOLD，执行层修复后重新提交。

5. 把 PR 链接发给 ChatGPT，让它做独立 PM Review。

6. 负责人验收，合入主分支。

核心原则就三条。执行 Agent 不自己审自己。每个 PR 只解决一个问题。高风险动作必须保留人工授权点。

不是所有任务都要走这六步。

低风险任务走快速通道。文案修正、测试期望修复、注释文档、单文件小改，执行层修改，提供最小证据，负责人确认，合入。四步完事。

但只要涉及数据库、调度、算法、跨模块、对外表达，不管任务多小，自动升级标准通道。这个硬边界不能破。

如果你也在用 AI 写代码，别急着搭复杂系统。先找一个低风险任务，按这条链路跑一遍。Issue 写清楚，PR 留证据，独立 Review，人工确认合入。

如果你暂时不想上完整六步，也可以先跑一个最小版本。第一步，Issue 写清楚目标、范围、禁区、验收命令。第二步，PR 只解决一个问题，附测试结果。第三步，代码复审和 PM Review 分开。第四步，负责人确认后再合入。

AI Coding 不是少写流程，而是把流程压缩到最小，但不能把责任链压没。

跑完一次，你就知道问题不在 AI 能不能写，而在你有没有给它一套可追责、可复盘、可验收的协作秩序。

本文基于实际多 Agent 协作工程实践整理。这套方案不是依赖某一个工具，而是借 GitHub 的 PR、代码审查和分支保护机制，把 AI 执行、AI 复审、AI 审计和人工合入拆开。后续我会把角色定义、流程图和任务包模板单独整理一版。
